# Content Development Plan — Process Guide

How to produce a content development plan for any week of Riipen Career Catalyst.

This document is a reference for generating development plans across all 15 weeks. It codifies the process used in the Week 3 exercise and generalizes it based on observed structural variation across weeks.

---

## Inputs

Each development plan requires one input:

- **The weekly design document** (`week[N]-design-doc.md` in project files)

The design doc contains everything needed: the activity breakdown (with activity numbers, categories, formats, and time allocations), the deliverable connection section, the AI guidance, and the open questions.

---

## Process

### Step 1: Identify the week's anchor(s)

Read the design doc's At a Glance and Deliverable Connection sections. Identify:

- **Anchor event(s):** The employer touchpoint, graded submission, or checkpoint that organizes the week. Most weeks have one anchor; some have two (e.g., a checkpoint + a reflection, or a graded deliverable + an employer meeting).
- **Reflection:** Does a DEAL reflection fall this week? (Weeks 3, 6, 9, 12, 15 only.) If yes, it is always a separate content piece — it never folds into another anchor.

The anchor determines the week's content architecture. Everything else in the week either prepares for it, follows from it, or is allocated time that doesn't need produced content.

### Step 2: Sort every activity into one of three buckets

Go through the activity breakdown row by row and classify each activity:

| Bucket | What goes here | Examples |
|---|---|---|
| **Absorbed into an anchor** | Module activities and project work steps that serve a specific anchor event or deliverable. These become sections or sub-elements within an anchor prompt. | Module content that teaches a skill needed for the deliverable; prep activities for an employer meeting; post-meeting debrief and summary steps |
| **Standalone content piece** | Activities that don't map to any anchor but still require produced content. Rare — most module content maps to an anchor. | A module on a transferable skill that isn't tied to this week's deliverable (e.g., a professional skill introduced for future use) |
| **No content needed** | Activities that are allocated time, not instructional content. | Async team communication, admin/submission tasks, unstructured project work blocks, team sync meetings |

**Decision rule:** If a module activity exists to help students do the anchor task better, it gets absorbed into the anchor prompt. If it teaches something students will need later but not this week, it's standalone. If it's just time for students to work, it's no-content.

### Step 3: Determine the content piece list

Based on the sorting, assemble the list of content pieces to produce. Every week will have at minimum:

1. **Week Overview** — always present, always absorbs the X.01 orientation activity
2. **Anchor Prompt(s)** — one per anchor event/deliverable; carries the instructional weight

Most weeks will also have:

3. **What's Next** — transitional brief bridging to the following week; absorbs any forward-looking activities

Some weeks will have:

4. **Reflection Prompt** — only in Weeks 3, 6, 9, 12, 15; always a separate piece
5. **Standalone Module(s)** — only when a module doesn't map to any anchor (uncommon)

### Step 4: Specify each content piece

For each content piece, document:

- **Piece title** — descriptive name (e.g., "Direction Check Anchor Prompt," not just "Anchor Prompt")
- **Type** — one of: Orientation text, Assignment prompt with embedded guidance, Transitional prompt, Reflection prompt, Standalone module
- **Absorbs** — list the activity numbers from the design doc that fold into this piece
- **Content outline** — what sub-elements this piece contains, organized by logical sections (e.g., before/during/after for a meeting-anchored week; research → synthesis → drafting for a production-anchored week)
- **Production approach for each sub-element** — one of:
  - `AI-generated text` — original text produced via AI (the default)
  - `Curated link + framing` — external resource with a one-sentence framing sentence (for generic professional skills)
  - `Template` — a reusable template students fill in (e.g., email template, agenda template, outline template)
  - `Checklist` — a scannable list students reference mid-task
  - `Interactive practice` — a short simulated scenario with interpretation questions (for pre-first-time experiences)
  - `AI-generated video` — short motivational/framing video via Synthesia or Lumen5 (for overviews and "why this matters" framing)
  - `Scenario card` — illustrated example showing what something looks like in practice

**Tagging convention for production approach:** Each sub-element should have its production approach noted in parentheses at the end of the line, e.g., `- How to frame your findings for a non-expert audience (AI-generated text; ~200 words)`.

### Step 4b: Sanity-check content against time allocations

Each activity in the design doc has a time allocation in minutes. The produced content for that activity — wherever it lands (absorbed into an anchor, standalone, etc.) — should be plausibly fillable within that time budget. This matters for accreditation: the time estimates need to hold up.

**How time breaks down for a module activity:**

- **Reading time** — the time spent reading produced content. Assume ~200 words/minute for scannable instructional text.
- **Thinking time** — legitimate processing, not reading. A 15-min module doesn't need 15 min of reading; students sit with the material, re-read sections, connect ideas. This is real time and doesn't need content to fill it.
- **Interactive/media time** — videos (~1–2 min each), interactive scenarios (~5–10 min each), and curated linked resources (~3–10 min depending on length) all eat into the time budget.
- **Doing time** — already accounted for in project work blocks ("no content needed" activities). Don't double-count it.

**The model for sizing content:**

Produced content (AI-generated text, templates, checklists) should be sized to what's actually needed — not padded to fill time. When the reading budget leaves room, use **curated links to external resources** to extend the time envelope rather than writing more text. This keeps produced content lean, follows Principle 3 (link, don't build for generic skills), and fills the time honestly. Not every module needs linked resources — sometimes focused text plus thinking time is exactly right. But when a topic has good external material, linking out is preferred over writing more.

**Rough guide (not a formula):**

| Activity time | Produced content target | Remaining time filled by |
|---|---|---|
| 5–10 min | ~200–500 words | Thinking time |
| 10–15 min | ~400–800 words | Thinking time; possibly a curated link or short video |
| 20–30 min | ~600–1,200 words | Thinking time + curated links and/or interactive elements |
| 40+ min | ~800–1,500 words | Curated links, interactive elements, thinking time |

These are starting points. Adjust based on content function — a template might be 200 words but take 20 min to fill in; a curated article might account for 5–10 min of reading without any produced word count.

**After specifying sub-elements in Step 4, scan the full content piece and check:**
1. Add up the estimated word counts, video runtimes, interactive element time, and curated link reading time.
2. Compare to the total time allocation of the absorbed activities.
3. If the total feels thin relative to the time budget, look for opportunities to add a curated link or deepen a sub-element — don't pad with filler text.
4. If the total exceeds the time budget, cut or consolidate.

### Step 5: Note what doesn't need produced content

List the activities classified as "no content needed" and briefly state why. This makes explicit that they were considered and deliberately excluded.

### Step 6: Write the production summary

A concise checklist at the bottom of the plan:

- Count of content pieces
- List of each piece with its type
- Any flags or dependencies (e.g., "Anchor prompt depends on rubric being finalized," "Curated links needed for X — requires link research")

## Content Development Principles

Apply these principles when making decisions within the process above. They are the design heuristics — when in doubt about how to handle an activity or structure a content piece, these resolve the ambiguity.

1. **One anchor prompt per week carries the instructional load.** Module content folds into the assignment prompt it serves rather than existing as standalone pieces.

2. **Recommend, don't prescribe for process steps.** "We recommend you debrief immediately after the meeting" embedded in context, rather than a standalone tracked activity.

3. **Link, don't build for generic professional skills.** Curate existing external resources with a framing sentence. Reserve original content production for course-specific material.

4. **Front-load "Why This Matters."** Every section opens with professional relevance — why a working professional would care — not learning objectives.

5. **Content lives at the point of need.** No standalone modules that exist only to prepare for something later. Embed guidance where students will use it.

6. **MVP production values.** AI-generated text and curated links as defaults. Original production only where course-specific and no adequate external resource exists.

7. **Use varied media to match the function.** Framing/motivation → short AI video. Instructional core/reference → scannable text. Scenario examples → illustrated cards or short animated scenarios. Optional/variable-need content → collapsible sections. **Scatter AI-generated videos across text-heavy weeks and pages to break up density.** Every "Why This Matters" opener is a candidate for video over text. Orientation weeks (Pattern E) and weeks with multiple standalone modules are especially prone to text fatigue — look for 2–3 additional video opportunities beyond the standard Week Overview video. The rule of thumb: if a section's job is to motivate or frame rather than instruct, it's a video.

8. **Active practice before high-stakes firsts.** Before any important first-time event (kickoff, direction check, progress review, final presentation), include a short interactive practice rep: simulated scenario → interpretation questions → debrief.

9. **Weekly structure: Overview → Anchor Prompt → What's Next → Reflection.** Repeatable pattern. Reflection only in Weeks 3, 6, 9, 12, 15.

## Structural Patterns

Not every week looks the same. Here are the recurring patterns and how to handle them:

### Pattern A: Single anchor event (meeting-centered)
**Example:** Week 3 (kickoff), Week 5 (Direction Check)

The week revolves around an employer touchpoint. Module content teaches skills needed for the meeting. The anchor prompt organizes as **before / during / after** sections. Most module activities absorb into the anchor.

### Pattern B: Single anchor deliverable (production-centered)
**Example:** Week 4 (Context Analysis & Project Plan)

The week revolves around producing and submitting a graded deliverable. Module content teaches analytical or production skills that feed directly into the deliverable. The anchor prompt organizes around the **production workflow** (research → synthesis → drafting → review → integration). Project work activities are no-content blocks; module activities absorb into the anchor.

### Pattern C: Production week with no major anchor
**Example:** Week 6 (first real production week), Week 8 (mid-production)

No single deliverable or event dominates. The module content may teach transferable skills (working through ambiguity, peer feedback, scope management). The anchor prompt is lighter — it may organize around the **production rhythm** for the week (what to focus on, how to manage your time, what "done" looks like this week). Some module content may be standalone if it doesn't serve a specific deliverable.

### Pattern D: Dual anchor (deliverable + employer meeting, or deliverable + reflection)
**Example:** Week 9 (Draft Deliverable + Reflection #3), Week 7 (WIP Summary + Progress Review #1)

Two things demand produced content. Handle each anchor as a separate content piece. The reflection is always its own piece. If there's a deliverable and a meeting in the same week, decide which one the module content primarily serves — that's the primary anchor, and the other gets a lighter treatment.

### Pattern E: Orientation/formation week
**Example:** Week 1 (course orientation), Week 2 (team formation)

No single deliverable or event dominates, but there's a lot of module content covering foundational skills. Multiple modules may be genuinely standalone because they're introducing skills for future use, not serving this week's deliverable. The anchor prompt may be thinner, and more content may stay as standalone modules — but still apply Principle 1 (fold into anchors where possible) before defaulting to standalone.

## Output Format

The development plan is a markdown file named `week[NN]-development-plan.md` (zero-padded). Structure:

```markdown
---
week: [N]
title: [Week Title]
created: YYYY-MM-DD
last_updated: YYYY-MM-DD
design_doc: week[NN]-design-doc.md
---

# Week [N] — [Title]
## Content Development Plan

---

## Content Architecture

[2-4 sentences describing: what organizes this week, how many content pieces
it reduces to, which pattern (A-E) applies, and what the Overview → Anchor →
What's Next → Reflection flow looks like for this specific week.]

---

## Content Production Plan

### 1. Week [N] Overview
**Type:** Orientation text
**Absorbs:** [N].01
**Format:** [production approach]

[Brief description of what the overview covers for this week.]

---

### 2. [Anchor Prompt Title]
**Type:** [type]
**Absorbs:** [activity numbers]

[Organized sub-elements with production approaches noted in parentheses.]

---

### 3. [What's Next / Additional pieces as needed]
...

---

## No Content Needed

[List of activities that are allocated time only, with brief rationale.]

---

## Production Checklist

[Checkboxed list of every production item, organized by page.]

[Roll-up sections: Links to Source, Linked Resources, Templates to Build,
Interactive Elements to Design, Scenario Cards to Create, Flags / Dependencies.]
```

## How to Use This in Claude Code

1. Open the weekly design document (`week[NN]-design-doc.md`)
2. Follow Steps 1–6 above, using the design doc as the sole input
3. Apply the Content Development Principles to resolve judgment calls
4. Match the week to the closest Structural Pattern (A–E) to determine the content architecture
5. Write the output in the format specified above
6. Save as `week[NN]-development-plan.md` with YAML frontmatter

When working in batches, process one week fully before moving to the next. Each plan should stand alone — no cross-references to other development plans are needed.
